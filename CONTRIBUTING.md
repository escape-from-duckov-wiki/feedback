# Contributing to Escape From Duckov Feedback

Thank you for your interest in contributing to Escape From Duckov! This document provides guidelines for submitting feedback and contributing to our community.

## üéØ Purpose of This Repository

This repository is specifically for:
- **Bug reports** - Technical issues with the website
- **Feature requests** - Suggestions for new features or improvements
- **Data updates** - Corrections or updates to game data
- **General feedback** - Questions, suggestions, and other feedback

This is **not** a repository for:
- Source code contributions (this is a feedback-only repository)
- General game discussion (please use appropriate gaming forums)
- Support requests for the game itself (contact the game developers)

## üìù How to Submit Good Feedback

### For Bug Reports

A good bug report should include:
1. **Clear title** - Briefly describe the issue
2. **Detailed description** - What went wrong?
3. **Steps to reproduce** - How can we see the bug ourselves?
4. **Expected vs actual behavior** - What should happen vs what did happen
5. **Environment details** - Browser, device, operating system
6. **Screenshots** - Visual evidence is very helpful

Example:
```
Title: [BUG] Search function returns no results on mobile Safari

Description: When using the search feature on iPhone, entering any search term
shows "No results found" even for items that definitely exist.

Steps to reproduce:
1. Open escapefromduckov.net on iPhone Safari
2. Click the search icon
3. Type "M4A1"
4. See "No results found"

Expected: Should show M4A1 rifle and related items
Actual: Shows no results
```

### For Feature Requests

A good feature request should include:
1. **Problem statement** - What problem does this solve?
2. **Proposed solution** - How would you like it to work?
3. **Use cases** - When would you use this feature?
4. **Priority** - How important is this to you?
5. **Examples** - Screenshots or examples from other sites (if applicable)

### For Data Updates

A good data update request should include:
1. **Specific item/location** - Exactly what needs updating
2. **Current (incorrect) data** - What's wrong now
3. **Correct data** - What it should be
4. **Source** - Where did you find the correct information?
5. **Evidence** - Screenshots, links to patch notes, etc.

## üîç Before Submitting

1. **Search existing issues** - Someone might have already reported it
2. **Check if it's still an issue** - Visit the website to confirm
3. **One issue per report** - Don't combine multiple unrelated issues
4. **Be respectful** - We're all here to help improve the site

After you submit an issue, maintainers will add labels:

- `bug` / `feature` / `data-update` / `other` - Issue type
- `priority-high/medium/low` - How urgent it is
- `in-progress` - We're working on it
- `resolved` - Issue has been fixed
- `duplicate` - Already reported
- `wontfix` - We've decided not to address this
